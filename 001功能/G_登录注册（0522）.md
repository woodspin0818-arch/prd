## 功能 TX-X001：PC端扫码/账号登录

---

### 1. 需求背景与目标 (Why)

PC端SaaS产品需要低摩擦的登录方式。利用微信公众号"带参二维码"能力，将扫码关注、账号绑定、登录三个动作合并为一个流程，消除传统注册中的密码设置、邮箱验证等流失点。同时保留账号密码登录作为兜底方案。

---

**账号状态说明**：

| 状态名称 | 状态含义 | 进入条件 | 可触达的状态 |
|----------|----------|----------|-------------|
| 待扫描 | PC端已生成二维码，等待用户扫码 | 页面加载/点击刷新二维码 | 已扫描、已过期 |

#### 3.4 后台管理：公众号带参二维码管理（选填）

> 本功能涉及的二维码由后端通过微信公众号API自动生成，运营人员无需手动管理。二维码参数规则如下：

| 配置项 | 说明 |
|--------|------|
| 二维码类型 | 临时带参二维码（QR_SCENE） |
| 有效期 | 600秒（10分钟），过期后PC端可手动刷新 |
| 场景参数 | scene_str = PC端生成的session_uuid（UUID v4，36位） |
| 回调处理 | 微信服务器推送XML事件到后端 → 后端解析scene_str → 通过WebSocket/长轮询通知对应PC端 |

#### 3.5 数据流（本功能相关）

| 数据项 | 来源 | 更新时机 | 缓存策略 | 前端存储 |
|--------|------|----------|----------|----------|
| 二维码图片URL | API-LOGIN-01 | 页面加载/手动刷新 | 不缓存 | 无 |
| session_uuid | 前端生成 | 页面加载/手动刷新 | 内存（页面级） | sessionStorage |
| 扫码状态 | WebSocket/长轮询推送 | 服务端主动推送 | 不缓存 | 内存 |
| 登录token | API-LOGIN-02 / WebSocket推送 | 登录成功后 | - | localStorage（HttpOnly Cookie优先） |

## 功能 TX-X002：公众号自动回复与引导

---

### 1. 需求背景与目标 (Why)

新用户关注公众号后，需要第一时间被引导至手机号绑定流程。利用微信公众号的"关注后自动回复"能力，实现零人工干预的注册引导。**已关注公众号但未绑定的用户**扫码后会触发SCAN事件（非subscribe事件），此时通过PC端提示+公众号关键词回复的方式完成引导。

### 2. 详细功能说明

#### 2.1 公众号自动回复配置

| 配置项 | 内容 |
|--------|------|
| 触发条件 | 用户关注公众号（subscribe事件） |
| 回复类型 | 图文消息 / 文字+超链接 |
| 回复文案 | "欢迎使用[SaaS产品名称]！点击此处绑定手机号，完成账号激活并登录PC端。" |
| 链接目标 | H5绑定手机号页URL，携带scene参数（从带参二维码的scene_str获取） |
| 链接有效期 | 与PC端二维码有效期一致（600秒） |

#### 2.2 交互流程

```
 用户关注公众号
       │
       ▼
 微信服务器推送 subscribe 事件 → 后端
       │
       ├── 事件中携带 EventKey（即带参二维码的scene_str）
       ├── 是否有对应的未完成PC会话？
       │     ├── 是 → 下发绑定手机号链接（携带scene）
       │     └── 否 → 下发普通欢迎语 + 菜单引导
       │
       ▼
 用户收到自动回复消息
       │
       ▼
 点击链接 → 打开H5绑定手机号页
```

#### 2.3 已关注用户扫码处理（SCAN事件）

已关注公众号但未绑定手机号的用户扫码时，微信推送的是 **SCAN事件**（非subscribe事件），不会触发自动回复。处理流程如下：

```
 已关注用户扫码
       │
       ▼
 微信服务器推送 SCAN 事件 → 后端
       │
       ├── 事件中携带 EventKey（即带参二维码的scene_str）
       ├── 查找用户是否已绑定手机号？
       │     ├── 已绑定 → 直接通知PC端登录成功
       │     └── 未绑定 → 通知PC端显示引导文案
       │                  │
       │                  ▼
       │           PC端二维码区域显示：
       │           "您已关注公众号，请在公众号内
       │            回复【登录】获取绑定链接"
       │                  │
       │                  ▼
       │           用户在公众号内回复"登录"
       │                  │
       │                  ▼
       │           后端通过客服消息下发绑定链接（携带scene参数）
       │                  │
       │                  ▼
       │           用户点击链接 → 打开H5绑定手机号页
       │
       ▼
  后续流程与subscribe事件一致

```

#### 2.4 公众号关键词回复配置

| 配置项 | 内容 |
|--------|------|
| 触发条件 | 用户发送关键词"登录" |
| 处理逻辑 | 后端查找该用户最近一次扫码的未完成PC会话 → 下发绑定链接 |
| 回复类型 | 客服消息（主动推送，不受48小时限制的场景：用户扫码/点击菜单） |
| 回复文案 | "点击此处绑定手机号，完成账号激活并登录PC端。" |
| 链接目标 | H5绑定手机号页URL，携带scene参数 |
| 链接有效期 | 与PC端二维码有效期一致（600秒） |
| 异常处理 | 无未完成PC会话时，回复"请先在PC端扫码后再回复【登录】" |

## 功能 TX-X004：密码重置（忘记密码）

---

### 1. 需求背景与目标 (Why)

新用户注册时设置了密码，后续可能遗忘。同时，H5设置密码流程中断（用户绑定手机号后关闭页面）的账号处于"待激活"状态，也需要通过密码重置来补设密码并激活账号。本功能提供短信验证码校验 + 重设密码的标准找回流程。

---

### 2. 详细功能说明

#### 2.1 PC端页面：密码重置页

##### 2.1.1 入口

| 入口 | 说明 |
|------|------|

## 模块汇总

> **核心说明**：
> 1. 登录注册的核心设计思想是"扫码即注册，关注即登录"——利用微信公众号带参二维码将PC端会话与手机端绑定流程串联，消除传统注册表单的流失点。
> 2. 双Tab设计（扫码/账号密码）确保兜底路径的可用性，同时扫码登录作为默认Tab体现产品的首选登录方式。
> 3. WebSocket为首选实时通信方案（低延迟），长轮询为降级方案（兼容不支持WebSocket的网络环境）。
> 4. 绑定手机号后必须设置密码才能进入系统，不可跳过，确保账号安全性。
> 5. H5页面需做微信环境检测，非微信环境下降级隐藏"微信授权获取手机号"按钮，引导用户走短信验证码流程。
> 6. 已关注公众号但未绑定的用户，扫码后通过PC端提示+公众号关键词回复（回复"登录"）完成绑定引导，与subscribe事件形成互补。
> 7. 新增PC端"注册账号"入口（TX-X005），为无微信/不愿扫码的用户提供手机号+短信验证码的独立注册路径。
> 8. 忘记密码流程（TX-X004）同时也是H5中断（绑定手机号后关闭页面）的恢复路径——通过密码重置激活"待激活"账号。

## ⚠️ 待确认事项

1. **WebSocket vs 长轮询优先级**：当前设计以WebSocket为首选、长轮询为降级。如果项目基础设施偏传统，是否需要以长轮询为主、WebSocket为后期优化？当前假设：WebSocket优先。
2. **密码复杂度要求**：当前设计不做强制复杂度要求（仅校验长度6-20位），以降低注册流失。是否需要在后期迭代中增加复杂度要求？当前假设：暂不强制。
3. **二维码有效期**：已确定600秒（10分钟），待实际运行后根据用户行为数据验证是否合适。
4. **多端扫码并发**：如果用户在PC端A打开了登录页（未扫码），又在PC端B打开了登录页——两个session_uuid是否都有效？当前假设：允许多端同时持有有效二维码，以先完成绑定的终端为准。

6. **临时token安全**：当前保持URL参数传递方式，有效期600秒。后期是否需要改为后端Session管理？当前假设：URL参数风险可控。
